# 📖 Описание проекта

Контакт-центры ежедневно обрабатывают тысячи обращений. Ручной контроль качества и проверка комплаенса — это дорого, долго и сложно масштабируется.

Данный проект представляет собой прототип аналитической ИИ-платформы, способной:
* Автоматически транскрибировать звонки (с использованием локальных ASR-моделей).
* Разделять речь по спикерам (Оператор / Клиент).
* Классифицировать тематику обращений и оценивать приоритет.
* Оценивать качество работы оператора по заданному чеклисту.
* Выявлять нарушения безопасности (комплаенс).
* Генерировать краткое резюме звонка и задачи.
* Интегрироваться напрямую в интерфейс чата OpenWebUI.

Весь стек контейнеризирован и запускается одной командой Docker Compose.

# ✨ Ключевые возможности

### 🎙 Локальное распознавание речи
* ASR на базе `faster-whisper` (модель `medium`).
* Оптимизировано для русского языка.
* Офлайн-обработка с сохранением временных меток.
* Поддержка множества аудиоформатов.

### 👥 Диаризация спикеров
* Разделение голосов на базе `pyannote.audio`.
* Назначение ролей: Оператор / Клиент.
* Структурированный вывод диалога.

### 🤖 Мульти-агентная аналитика (LangGraph)
Четыре специализированных ИИ-агента работают в детерминированном StateGraph:

| Агент                | Зона ответственности                                      |
| -------------------- | --------------------------------------------------------- |
| 🏷️ Классификатор      | Определение темы звонка и оценка приоритета.              |
| ⭐ Агент качества     | Оценка работы оператора по чеклисту.                      |
| 🛡️ Комплаенс          | Проверка безопасности и выявление запрещенных фраз.       |
| 📝 Суммаризатор       | Составление краткого резюме и списка дальнейших действий. |

### 🌐 REST API
* Бэкенд на FastAPI.
* Прием аудиофайлов через `multipart/form-data`.
* Интерактивная документация Swagger.

# 🧠 Обоснование стека технологий

### Почему LangGraph? (StateGraph)
В проекте используется последовательный граф: `Классификатор → Качество → Комплаенс → Суммаризатор`.
Каждый аналитический шаг зависит от одного и того же транскрипта и выполняется ровно один раз. Последовательный `StateGraph` обеспечивает детерминированность, снижает нагрузку на LLM по сравнению с паттерном Supervisor (где агент-маршрутизатор может ошибиться) и идеально подходит для данной задачи.

### LLM: Qwen 2.5 7B Instruct
Используется через OpenRouter. Выбрана за сильное понимание русского языка, четкое следование инструкциям, генерацию валидного JSON (structured output) и оптимальное соотношение цены и качества.

# 💬 Интеграция с OpenWebUI

Реализован нативный кастомный Pipeline (`pipeline.py`) для работы с платформой непосредственно через интерфейс чата.

**Поддерживаемые методы передачи аудио:**
1. **Загрузка через интерфейс (кнопка-скрепка):** Пользователь прикрепляет аудиофайл прямо в чат. Пайплайн перехватывает внутренние XML-теги `<attached_files>` системы OpenWebUI, извлекает ID файла, находит его физический путь на жестком диске контейнера, динамически определяет MIME-тип (wav, mp3, m4a) и отправляет в бэкенд. Это исключает проблемы с deadlock при внутренних HTTP-запросах.
2. **Передача прямой ссылки (URL):** Пользователь отправляет HTTP/HTTPS ссылку на файл обычным текстовым сообщением. Пайплайн находит её через регулярные выражения, скачивает аудиофайл во временную директорию и отправляет на анализ.

В чат в реальном времени транслируются статусы обработки, после чего выдается красиво отформатированный Markdown-отчет.

# ⚙️ Установка и локальный запуск

1. Клонируйте репозиторий:
```bash
git clone https://github.com/Carcajo/mtbank-ai-call-analytics.git
cd mtbank-ai-call-analytics
```
2. Создайте файл .env из примера:
```bash
cp .env.example .env
```
3. Укажите ваши ключи HF_TOKEN (для PyAnnote) и OPENROUTER_API_KEY (для LLM).

4.Запустите через Docker:
```bash
docker compose up -d --build
```
(Первый запуск может занять несколько минут из-за скачивания моделей Whisper и PyAnnote).ько минут из-за скачивания моделей Whisper и PyAnnote).
